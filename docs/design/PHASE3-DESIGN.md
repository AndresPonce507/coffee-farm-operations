# Janson Coffee — Phase-3 · DESIGN (Architecture) — "Monetize the chain: contract-to-cash, bag-to-cup, books-that-reconcile-themselves"

> **This document is the Phase-3 SSOT contract DISTILL/DELIVER build against.** It decomposes Phase 3 into **P3-S0 … P3-S20** with the same spine-first ordering rule the Phase-1 and Phase-2 plans used: each slice = one migration (or none) + its `src/lib/db/*` ports + a world-class liquid-glass UI surface + failing-first tests + a dogfood moment, with the cross-slice rails carried forward from Phase 1/2. Phase 3 turns the `green_lots` + cup scores the make-quality trunk produces into priced, contracted, shipped, documented, and sold coffee — without ever letting a $30k/kg Best-of-Panama Geisha anchor to the commodity index, oversell a microlot, ship a doc without its prerequisites, conjure mass in the mill, or leak PII/cost on the public web. ### C. Multi-tenant timing (per-slice posture + the recommendation)

**Every Phase-3 slice today must ship authenticated-only RLS (`using (true)` to `authenticated`, append-only tables add the insert policy, nothing to `anon` except P3-S13) — because that is the only posture the shipped spine supports.** There is no `current_farm_id()`/`current_tenant_id()` to reference; a Phase-3 migration that adds `farm_id … default current_farm_id()` will fail to apply. **Recommendation (matches the Phase-4 doc's resolved decision): land the multi-tenant substrate (P4-S0: `tenants`/`tenant_members`/`current_tenant_id()` + `tenant_id` on every table + the `using (tenant_id = current_tenant_id())` rewrite) EARLY — ideally as the first migration after phase 2 and BEFORE the bulk of Phase 3 — so every Phase-3 table is born tenant-scoped instead of being retrofitted.** The cost asymmetry is decisive: retrofitting `tenant_id` onto ~20 Phase-1 tables PLUS all of Phase 2 PLUS all of Phase 3 (pricing/contracts/orders/AR/CRM/provenance — the most sensitive PII+money tables in the program) multiplies the missed-policy-silent-leak risk. **If P4-S0 lands before Phase 3:** every Phase-3 table carries `tenant_id` + the tenant policy from line one, every command RPC self-clamps `tenant_id := current_tenant_id()`, and the P3-S13 anon door must additionally filter `tenant_id` in `resolve_provenance`. **If Phase 3 ships first (current plan):** it ships authenticated-only and P4-S0 retrofits it — acceptable but more rework. **Net: rail #10 is not "dormant `farm_id`" — it is "ship authenticated-only now; tenant-ize via P4-S0, and prefer landing P4-S0 before Phase 3 to minimize retrofit."**

### D. The "P2-S0 outbox" does not exist — and other phase-2 name corrections

- **There is NO offline outbox / `enqueueCommand()` / `dispatch_outbound` substrate anywhere in the 5 phase-2 migrations.** Every phase-2 (and phase-1) command RPC *accepts* `device_id`/`device_seq`/`idempotency_key` so it is offline-replayable in shape, and phase-2 adds `worker_server_seq` + `next_server_seq()` (a server seq minter) — but the client-side IndexedDB queue + replay engine the doc calls "the P2-S0 outbox" **was never built**. ## §1 — Cross-slice rails (carried from Phase 1/2; every Phase-3 slice obeys these)

These are the invariants and conventions that hold across the whole phase. A slice never re-derives them; it inherits them.

1. **One write door.** All writes go through `SECURITY DEFINER` command RPCs, `set search_path = public, extensions`, idempotent on a client-minted `idempotency_key` (+ `device_id`/`device_seq` for field surfaces), appending the relevant `lot_event` in the *same* transaction. No client `UPDATE`/`DELETE` grant on any ledger; corrections are reversing/superseding rows. (The Phase-1 `record_cherry_intake`/`materialize_green_lot`/`cost_entry` posture.)
2. **AD-8 grants, exactly.** Per-object `grant select … to authenticated` on every table/view (one statement per object so the name-anchored static guard matches); `revoke execute … from public` **then** `grant execute … to authenticated` on every RPC; `anon` gets nothing except the single P3-S13 public provenance projection.
3. **Append-only, hash-chained audit.** Every commercial decision (price quoted/accepted, fixation locked, contract signed, sample sent, export doc issued, shipment departed, cash received, accolade awarded) appends to the Phase-1 `lot_event` hash chain via `record_lot_event`, keyed `stream_key` = the green lot's `code`/`lot_code` (or doc number). `verify_chain(lot_code)` therefore covers a lot's *entire commercial life*, not just its agronomy. New PII-bearing event ledgers (`contact_events` in P3-S18, `lot_accolades` in P3-S19) reuse the **generalized `event_set_hash()` util** — extracted *additively* in P3-S18 (the existing `lot_event_set_hash` delegates to it; the live chain keeps passing `verify_chain`), never an in-place rewrite of the shipped hash logic.
4. **The money guarantee is reused, not rebuilt** (§0.2). The ATP trigger family is the single oversell seam.
5. **Margin floor / cost truth.** Every price reads `cogs_per_lot(lot)`/`mv_lot_cost.cost_per_kg_green` (Phase-1 costing). Milling/roasting *post* `cost_entry` rows so processing cost flows into `cogs_per_lot`. Accounting derives `farm_season_config.ytd_revenue_usd` from `Σ revenue_entry` (closing the modeled-revenue gap the season-config hardening migration flagged).
6. **Conversions route through `convert_qty`/`units`.** The lb↔kg factor in the commodity "C" math is **never** a hardcoded `2.2046` in JS — it routes through `convert_qty` (roadmap gap #10, the named silent-corruption trap). USD↔settlement FX routes through the canonical `fx_rate` table (P3-S16), never an off-book rate.
7. **No untrusted inbound drives a write** (the global injection invariant). An inbound buyer email / WhatsApp reply / unsubscribe click / a customer's storefront text is *recorded* as evidence (a `contact_event`, a webhook row) but never autonomously fires a command RPC. AI may *draft* offer/campaign copy from real rows; a human clicks send. Money-shaped writes (settle payment, issue fiscal invoice, lock fixation) are human-confirmed, never auto.
8. **World-class liquid-glass in every slice** (docs/ROADMAP.md "Build principles"): real `backdrop-blur` only on floating chrome, glass-lite content cards, GPU-only transforms, `prefers-reduced-motion`-aware, WCAG-AA on glass, mobile-first, loading/empty/error states, brand forest `#00291D`. Spanish-first (es-PA) on buyer- and field-facing surfaces (~90% Ngäbe-Buglé crew on the POS). Every UI PR ships a jsdom render/smoke test; every logic RPC ships a failing-first PGlite DB test (the every-PR-ships-a-test, no-UI-exemption rule).
9. **Offline via the P2-S0 outbox.** Field-side and POS writes route through the Phase-2 offline outbox (`enqueueCommand()`), replaying exactly-once on the `idempotency_key`. Legal/money instruments (sign contract, issue export doc, settle payment, lock fixation) are online-first (they need server truth); read surfaces are cacheable.
10. **`farm_id` tenant seam, dormant.** Every new table carries `farm_id uuid not null default current_farm_id()` + an RLS `using (farm_id = current_farm_id())` policy now, so the Phase-4 multi-tenant license to neighboring Volcán estates is costless to retrofit and never reads another farm's rows.

## §4 — The key data-layer invariants (the phase's load-bearing guards)

1. **Fixation/contract/invoice/sample/allocation cannot oversell green inventory** — every commit inserts a `lot_reservations`/`lot_shipments` (and, post-S18, `sample_dispatches`) row; the Phase-1 `prevent_oversell` advisory-lock trigger is the single guard. A $30k/kg Geisha can't be double-sold across pricing, B2B, auctions, DTC, subscriptions, or AR. (S0, S1, S3, S4, S12, S17, S18.)
2. **A reserve lot can never anchor to the "C" index** — `enforce_regime_pricing` trigger + `price_quotes`/`green_offers` CHECKs reject `regime='commodity'` for a Presidential/Specialty single-origin (cup ≥ 88); the reserve-mandatory band is tunable in `farm_season_config`. (S0, S1.)
3. **An export/fiscal doc can't issue without its prerequisites** — `issue_export_doc`/`issue_ar_doc` raise unless the **declarative `export_doc_prereqs`** (evaluated against live state, calling Phase-1 `eudr_lot_status`) is clear; the B/L is chain-locked until the other four issue (UNIQUE partial index); a `dgi_factura` can't be 'issued' until the PAC stamped a CUFE. (S3, S17.)
4. **Closed outturn mass balance** — `finalize_milling_run` raises unless `green_out + Σbyproduct + accounted_moisture_loss ≈ parchment_in` within a `lot_yield_curve`-sourced ceiling; green is minted via `materialize_green_lot`'s conserved 'process' edge and byproducts via conserved 'byproduct' edges, so the Phase-1 `lot_edges_conserve_mass()` physically prevents the mill from conjuring or losing mass. A roast can't claim green it didn't consume (the conserved 'roast' edge). (S8, S9, S10.)
5. **Public provenance leaks nothing** — the anon door (`resolve_provenance` + `sku_provenance_public`, the only anon grant in the phase) reads ONLY whitelisted columns behind the `is_published` curation gate; a **static guard test pins the anon surface to exactly the whitelisted set** — any widening fails CI. No `workers.phone`/`daily_rate_usd`, no `cost_entry`, no `lot_reservations`/buyer, no unpublished SKU ever reaches the web. (S13.)
6. **Ledgers are append-only + immutable + hash-chained** — `revenue_entry`/`ar_payment`/`fx_gain_loss_entry`/`contact_events`/`lot_accolades`/`fg_ledger`/`sub_allocations` reuse the `cost_entry_immutable`/`lot_event` posture; corrections are reversing/superseding rows; sync is exactly-once (`sync_outbox` content-hash idempotency + `FOR UPDATE SKIP LOCKED`); a falsified revenue journal or a double-posted QBO entry is impossible by construction. (S16, S17, S18, S19.)
7. **Consent is enforced at the data layer** — the marketing send queue inserts only for consenting, non-unsubscribed contacts (a CHECK + outbound-queue guard, not a UI toggle). (S20.)

## §6 — The biggest open DECISION

**The fiscal-invoicing posture: how far to build the DGI Factura Electrónica seam before the family contracts a certified PAC — and whether domestic POS/B2B fiscal sales are even in Phase-3 scope.** Every other paid-service gate has a clean, genuinely-$0 shipping path (manual "C" marks, manual mill/storage readings, Artisan .alog, web-share marketing, Stripe-until-revenue, internal non-fiscal POS recibos, GS1 internal identifiers). The PAC is the one *unavoidable* paid dependency for a *legal* domestic fiscal invoice in Panama — and it's genuinely deferrable because most of Janson's revenue is USD export (which needs only the free commercial *trade* invoice, not a DGI factura). The decision: **(A)** ship the full `sync_outbox` target='dgi_pac' seam + the `dgi_cufe` columns + a stubbed dev adapter now, and treat the real PAC contract as a post-revenue flag the owner flips — OR **(B)** descope domestic fiscal invoicing from Phase 3 entirely, ship only the USD-export commercial-invoice path, and leave the DGI seam to a small Phase-3.5 once domestic café/farm-store volume actually exists. The recommendation is **(A)** — the seam is cheap to build, the stubbed adapter dogfoods end-to-end at $0, and it avoids a retrofit — but the family's actual domestic-sales volume (do they sell enough at the Lagunas café / farm store to justify a per-document PAC bill now?) is the input only Andres has. **This is the one place the plan needs a human decision before S14/S17 land.** (Secondary, smaller open question: whether the B2B-area `sample_dispatches` ATP-claim and the CRM-area `sample_dispatches` are one table or two — resolved in the S2↔S18 reviewer pass, leaning to one shared table owned by S18 with S2 logging through it.)
