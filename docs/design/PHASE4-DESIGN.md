## 1. The spine-first ordering rule (Phase 4 has THREE foundations, not one)

Phase 4 builds **no new operational trunk** — it is reads-and-intelligence over the now-complete Phase-1/2/3 spine. But three things are *foundational substrates* that everything else routes through, and the dependency graph forces a strict landing order on them:

1. **The tenant boundary (P4-S0).** Every other Phase-4 engine (copilot grounding, reserve detections, sustainability evidence, sensor readings, tree census, tourism bookings) writes new tables. **If the tenant boundary lands AFTER them, every one of those tables must be retrofitted with `tenant_id` + a tenant-scoped policy — doubling the RLS rewrite.** The multi-tenant slice is therefore not "the last Phase-4 slice" as the Phase-2 outline §6 implied; its *substrate* (`tenants`, `tenant_members`, `current_tenant_id()`, and the rule "every new table carries `tenant_id` + a `tenant_id = current_tenant_id()` policy from line one") **must land first**, so every subsequent Phase-4 table is tenant-aware at birth. (The full 20-table retrofit of the *existing* Phase-1 tables is the large back-half of S0; the substrate is the cheap front-half.) **Resolved decision: build-tenant-aware-now, not retrofit-later — see §4.**

2. **The human-in-the-loop proposal spine (P4-S1).** The copilot, the reserve's detection-verify gate, the agronomy recommendation gate, and the carbon-claim publish gate all express the *same* invariant: an automated/ML output is a *proposal* that only a human-confirm RPC can execute. The copilot slice builds the canonical version of this spine (`copilot_write_proposal` + `propose/confirm/reject_copilot_write` + the `model_run` citation contract + the `copilot_readonly` SELECT-only role). The reserve, agronomy, and sustainability slices each build their *domain-specific* confirm gate (`verify_species_detection`, `decide_recommendation`, `publish_carbon_claim`) — same posture, domain-local table. S1's proposal spine is the **reference implementation** the others clone; landing it first means the injection-invariant probe is proven once, early.

3. **The evidence-graph substrate (P4-S2 / the sustainability ledger).** The reserve engine's BirdNET counts, the milling-water L/kg, the per-tree shade census, and the cert dossiers all want to write into ONE append-only `sustainability_evidence` ledger rather than four private tables. So the **evidence ledger wants to land before the reserve and sensor slices that feed it** — but the reserve engine also produces the biodiversity rows the evidence graph reads. The clean resolution (designed below): the evidence ledger is **upstream substrate** (lands in S2), the reserve engine **writes INTO it via `record_evidence`** rather than a private table, and the sustainability *projections* (carbon claim, cert dossier) can be built late. One-way edges, no cycle.

**Must-land-first, in strict order:** **P4-S0 (tenant substrate + RLS retrofit) → P4-S1 (copilot proposal spine + injection probe) → P4-S2 (sustainability evidence ledger substrate).** Only after S0 is every later table born tenant-scoped; only after S1 is the human-confirm pattern proven and reusable; only after S2 does the reserve/sensor/cert work have one ledger to write into. Everything else fans out from there.

Composite index `<t>(tenant_id, <existing leading key>)` per table.
- **`lot_event` canonical-bytes change:** `lot_event_canonical_bytes` is extended to **prepend `tenant_id`** so a tenant cannot forge/replay into another's chain; `(tenant_id, stream_key)`-unique on `device_seq`; `verify_chain` recomputes per-tenant.
- **`mv_lot_cost` / `mv_lot_cost_by_rule`** gain a `tenant_id` column carried through `green_lot_mass` → `cogs_per_lot`/`cogs_per_plot`. **These matviews are owner-materialized — RLS does NOT apply to them** (the one structurally dangerous spot, flagged hard below).

**The claim-resolution keystone (one function the whole RLS rewrite leans on):**
```sql
create function current_tenant_id() returns uuid language sql stable security definer
  set search_path = public as $$
  select (auth.jwt() -> 'app_metadata' ->> 'tenant_id')::uuid $$;
```
reads the tenant claim Supabase Auth stamps into **`app_metadata`** (NOT user-settable, unlike `user_metadata` — that's the security boundary). Plus `is_tenant_member(p_tenant uuid)` / `has_role(p_role text)` reading `tenant_members`. AD-8 grants.

**The RLS rewrite** (a `do$$`-loop over the table array, the same mechanical pattern the existing `20260620140000_auth_required_rls.sql` uses, extended to all 20 tenant-scoped tables): replace every `using(true)` SELECT policy with `using (tenant_id = current_tenant_id())`; replace write-grant `with check (true)` → `with check (tenant_id = current_tenant_id())`. `lot_event` keeps `force row level security` + append-only, now also tenant-filtered.

**Command-RPC retrofit** (every existing `SECURITY DEFINER` writer — `record_cherry_intake`, `materialize_green_lot`, `advance_processing_stage`, `record_lot_event`, `eudr_declare_plot`, `refresh_lot_cost`): a definer fn **bypasses RLS**, so each must self-clamp — `v_tenant := current_tenant_id(); if v_tenant is null then raise 'no tenant'; end if;`, stamp `tenant_id => v_tenant` on every INSERT, add `and tenant_id = v_tenant` to every UPDATE/lookup WHERE. **`cogs_per_lot`/`cogs_per_plot` MUST add an explicit `and tenant_id = current_tenant_id()` against the matview** — the one place RLS does not auto-protect.

**Onboarding RPCs:** `provision_tenant(slug, name, prefix, owner_user)` (platform-operator-only, guarded by a `platform_operator` flag — creates tenant + owner membership + `lot_counters` seed); `accept_invitation(token)` (binds `auth.users.id` → `tenant_members`; an Edge Function stamps `app_metadata.tenant_id` since RLS can't write the JWT directly); `set_active_tenant(tenant)` for a future multi-tenant user (the platform operator / a bookkeeper serving two estates).

**Auth wiring (UI ports):** extend `src/middleware.ts` + `src/lib/supabase/middleware.ts` (`updateSession`) — after `getUser()`, read `app_metadata.tenant_id`; a signed-in user with NO claim redirects to `/onboarding/join` (the single chokepoint where tenant context enters every request). `src/lib/supabase/server.ts` gains a cached `getActiveTenant()`. An Edge Function stamps the claim on invitation accept.

**UI:** `/settings/team` (the roadmap's "visual matrix of people × roles × scopes" — members table, invite-by-phone, role/scope editor, `expires_at` badges); `/onboarding/join` (phone-OTP / magic-link accept, Spanish-first from `tenants.locale`); `/platform` (platform-operator-only tenant list + provision-new-estate wizard, hidden behind the flag). **Tenant branding:** a Server Component reads `tenants.branding` and themes the shell via CSS custom properties — Janson's forest **#00291D** is the DEFAULT, each estate overrides; the shared shell is read-not-forked.

**Key invariants (data-layer — defense-in-depth like the existing oversell/mass-conservation triggers):**
1. **Tenant read isolation** — `using (tenant_id = current_tenant_id())` on every operational table; a stolen anon key still sees nothing (anon has zero grants).
2. **No cross-tenant write** — INSERT `with check (tenant_id = current_tenant_id())` + definer RPCs self-stamp from the JWT claim, never a parameter (a definer fn bypasses RLS — this is the *only* thing stopping a crafted RPC writing into another estate; the exact S3 "PUBLIC execute let anon mint lots" lesson).
3. **No cross-tenant lot edge** — `lot_edges` mass-conservation trigger gains a guard: parent and child `lots.tenant_id` must match, else `raise 'cross-tenant lot edge'` (stops Estate A blending mass out of Estate B's lot — would also corrupt mass conservation across the boundary).
4. **No cross-tenant claim** — the reservation/shipment `tenant_id` must equal the green lot's `tenant_id` (trigger reading `lots`), extending the `prevent_oversell` family (stops Estate A reserving Estate B's scarce Geisha).
5. **Hash-chain tenant binding** — `lot_event_canonical_bytes` prepends `tenant_id`; `(tenant_id, stream_key)` uniqueness; `verify_chain` recomputes per-tenant.
6. **COGS tenant filter** — the explicit `and tenant_id = current_tenant_id()` against the un-RLS-able matviews (flagged hard).
7. **NULL-claim fail-closed** — `current_tenant_id()` returning NULL means every policy evaluates `tenant_id = NULL` = zero rows, and every definer RPC raises. A tokenless/un-onboarded user sees NOTHING.

**A STATIC GUARD SCRIPT** (the AD-8-guard discipline): parse the migration, assert EVERY operational table has both a `tenant_id` column AND a `tenant_id = current_tenant_id()` policy, and EVERY definer RPC references `current_tenant_id()`. Fails `npm run test` if a table is missed — a dead guard is itself an incident.

**Dependencies:** Phase-1 — the 20 tenant-scoped tables, the `auth_required_rls` `do$$`-loop pattern (reused verbatim, extended), the SECURITY DEFINER RPC roster (each retrofitted to self-clamp), `lot_code_seq` → `lot_counters`, the `lot_event` hash chain, `mv_lot_cost`/`mv_lot_cost_by_rule`, `grant_hygiene`'s locked defaults. Phase-2/3 — every Phase-2 and Phase-3 table SHOULD already carry `tenant_id` if S0's substrate landed as the first Phase-2 migration (see §4); if not, they are retrofitted here too.

**Dogfood moment:** provision a SECOND tenant alongside Janson — a tiny synthetic "Finca Vecina" (slug `finca-vecina`, prefix `FV`, one owner login, 2 plots, 1 green lot). Sign in as the FV owner: the dashboard shows ONLY FV's 2 plots and `FV-700` — Janson's 564 lots, the 200-ha quetzal reserve, the COGS, the EUDR dossier, the buyer book are all invisible. Sign back in as the Janson principal (Miguel/Janette): Finca Vecina has vanished. Run `verify_chain` for both — both return true independently. Attempt (via raw RPC) to reserve FV's green lot from Janson's session — the oversell+tenant trigger raises `cross-tenant claim`. **That is the moment the single-estate tool becomes a licensable platform** — hard isolation proven at the data layer, not the UI. It is also the live demo for the "Cropster for estates that own their whole chain" pitch to the Best-of-Panama peer estates (Esmeralda, Elida, Lamastus, Ninety Plus).

**Highest risk + de-risk:** a SILENT cross-tenant LEAK — one missed policy/view or a definer RPC that forgets to self-clamp leaks one estate's payroll/buyer-book/scarce-Geisha pricing to another. Catastrophic; exactly the TradelyHQ "tenant-clamp bug" class. Two structurally-dangerous spots: (1) the 2 matviews (RLS does not apply — must be filtered in the RPC, easy to forget); (2) any future table added without `tenant_id` (default-RLS-off in Postgres). **DE-RISK:** **(P4-SPIKE-S0, nw-spike, timeboxed)** prove the claim path end-to-end on ONE table — stamp `app_metadata.tenant_id` via Supabase Auth, read it in `current_tenant_id()`, confirm a 2-tenant PGlite test isolates `plots` — validating the JWT-claim mechanism (the one new external dependency) before committing to all 20 tables. Then the static guard script + a permanent cross-tenant probe test (two seeded tenants, assert zero row bleed across all 20 tables + all read RPCs + both matviews) + the phased gate + a reviewer pass closing the fan-out. **Migration safety:** non-destructive (add nullable `tenant_id` → backfill Janson uuid → set not null → swap policies in one txn); post-backfill the Janson session sees identical data (zero behavior change for the existing single tenant). The `using(true)`→`tenant_id=…` policy swap is the irreversible-feeling step — **flag + wait for explicit OK before applying.**

**Paid:** $0-compatible. Supabase Auth custom JWT claims via `app_metadata` + Postgres RLS — free tier, the exact mechanism the roadmap names. Branding/locale = data in `tenants.branding`. **Flagged (only when go-to-market activates, NOT for the architecture):** Stripe billing for `tenants.plan` (blessed exception, $0-until-revenue — wire it dormant); **WhatsApp Business Cloud API / Twilio Verify for phone-OTP is NOT free at scale → $0 fallback is Supabase email magic-link** (sufficient for Janson-internal dogfood); Auth0/Clerk SSO/SCIM is an enterprise feature, explicitly deferred. **The real $0 consideration is operating SaaS at all** (an SLA, support burden, recurring bill the moment tenant #2 onboards) — this slice builds the ARCHITECTURE only; the business decision to run multi-tenant infra stays dormant per the roadmap's "don't sell for 2-3 seasons."

## 4. The biggest open DECISION the family / Andres must make before building

**THE PRIMARY DECISION — multi-tenant timing: retrofit-later vs build-tenant-aware-now. RESOLVED: build-tenant-aware-now.** This is the single largest architectural call in Phase 4, and the dependency analysis forces the answer rather than leaving it open:

- **The cost asymmetry is decisive.** The Phase-1 spine ships `using(true)` with zero claim plumbing (confirmed by grep — no `auth.uid()`/`app_metadata` anywhere). Every Phase-4 engine — copilot grounding, reserve detections, sustainability evidence, sensor readings, tree census, tourism bookings — adds new tables. If the tenant boundary lands *after* them, all of those tables (plus every Phase-2/3 table) must be retrofitted with `tenant_id` + a tenant-scoped policy — **doubling the RLS rewrite and N-multiplying the risk of a missed-policy silent leak.** The roadmap's own words: *"the data model decisions are nearly free to get right now and ruinously expensive to retrofit."*
- **The resolution (two-part):** (1) the **substrate** — `tenants` + `tenant_members` + `current_tenant_id()` + the rule "every new table carries `tenant_id` + a `tenant_id = current_tenant_id()` policy from line one" — **should have landed as the FIRST Phase-2 migration** and, failing that, lands as **P4-S0's front half**; (2) the **full 20-table retrofit of the existing Phase-1 tables** is P4-S0's back half, the large gated slice. Critically, **the ARCHITECTURE lands immediately; the GO-TO-MARKET stays dormant** (the roadmap's explicit "don't sell for 2-3 seasons"). Building the boundary is not the same as operating SaaS — the business decision to run multi-tenant infra (an SLA, a support burden, a recurring bill, the moment tenant #2 onboards) is a *separate, deferred* call. We pay the cheap schema cost now to avoid the ruinous retrofit later, and we do NOT incur any infra bill by landing the schema.
- **What the family must still confirm before S0 applies:** the **role model** — single shared owner-login (simplest, but the supervisor uses the owner's session and per-actor accountability is lost) vs per-actor JWT-claim roles (real attendance/payroll attribution). **Recommendation: build the per-actor data layer (the `role`/`worker_id` claim path) but let the UX start single-device** (badge-to-act), growing into per-device logins — because attendance and payroll evidence is worthless if anyone can write as anyone. And the **Apply-OK on the policy swap itself** (`using(true)` → `tenant_id=…` touches every table — an auth change that always flags and waits).

**Secondary, flagged, must-confirm-before-the-dependent-slice:**
1. **Emission-factor values + the VM0042 sequestration formula (S2 — climate accounting, legal/reputational).** The IPCC/Cool Farm factor values and the agroforestry-inset math must be **confirmed by the family or an agronomist** and entered as versioned `emission_factor` data flagged `'PLACEHOLDER — agronomy review'`, gating any carbon claim leaving 'draft'. Wrong = a falsifiable/greenwashed claim a buyer's auditor catches.
2. **The Vercel AI Gateway cost ceiling (S1 — $0 vs metered).** The NL-query / answer-drafting LLM is the one external dependency; ship local-small-model-first + prompt caching + hard `model_run.cost_usd` caps + tier routing, and the SELECT-only role means even a cheap model can't harm. The NL feature degrades gracefully (disabled; forecasts/sell-vs-hold keep working on local deterministic methods). Confirm the per-month cap before enabling Sonnet-tier on demand.
3. **On-device pest-vision model quality (S1 — hardware reality).** The quantized leaf-classifier must work on the crew's cheap Android phones; the de-risk is the "uncertain" class always routing to a human-confirmed `pest_observation`. Confirm via a field test before trusting auto-classification.
4. **AudioMoth hardware purchase (S3 — one-time spend).** The bioacoustic source is additive; the whole reserve engine ships $0 on the manual eBird path. The family decides if/when to buy AudioMoth units (~$70 each, one-time) — not a blocker.
5. **The `rz-quetzal` real boundary (S3 — data gate).** The seeded reserve polygon is a documented PLACEHOLDER; a real traced boundary is a family/field gate before any in-reserve spatial claim is trusted.
6. **Phase-3 storefront/Stripe must exist before S5's attribution half (sequencing).** The booking + deposit half can ship first; the taste→QR→purchase loop waits on the live Phase-3 storefront SKU route.
